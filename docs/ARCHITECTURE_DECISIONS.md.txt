# ARCHITECTURE_DECISIONS.md

# OSINET Time ERP Enterprise
## Registro Oficial de Decisiones de Arquitectura

---

# PROPÓSITO

Este documento explica las decisiones de arquitectura del proyecto.

No describe cómo funciona el código.

Describe POR QUÉ el proyecto fue diseñado de esta manera.

Toda IA debe leer este documento antes de modificar la arquitectura.

---

# DECISIÓN 001

## Arquitectura MVVM

Se utiliza MVVM porque:

- separa responsabilidades
- facilita el mantenimiento
- permite escalar el ERP
- facilita las pruebas
- evita lógica dentro de Compose

Nunca mover lógica de negocio a las pantallas.

---

# DECISIÓN 002

## Room Database

Se utiliza Room porque:

- funciona sin Internet
- permite uso offline
- mantiene integridad de datos
- es estable para Android

No utilizar Firebase como almacenamiento principal.

---

# DECISIÓN 003

## Repository Pattern

Toda entidad Room debe tener Repository.

Nunca acceder directamente al DAO desde la interfaz.

Flujo obligatorio:

UI

↓

ViewModel

↓

Repository

↓

DAO

↓

Room

---

# DECISIÓN 004

## Engine Layer

La carpeta engine contiene reglas de negocio.

Ejemplos:

VacationEngine

PayrollEngine

AttendanceEngine

PdfEngine

WhatsappEngine

Los Engine NO reemplazan Repository.

Los Engine procesan información.

---

# DECISIÓN 005

## Compose

Compose solamente muestra información.

Nunca colocar cálculos empresariales dentro de una pantalla.

Toda lógica debe vivir en:

Engine

o

ViewModel

---

# DECISIÓN 006

## Navegación

Toda navegación pasa por AppNavigation.kt.

No crear NavHost secundarios sin necesidad.

Mantener una navegación centralizada.

---

# DECISIÓN 007

## Componentes reutilizables

Todos los módulos deben reutilizar:

OSINETButton

OSINETTextField

OSINETCard

OSINETTopBar

OSINETDialog

Evitar duplicar componentes.

---

# DECISIÓN 008

## Eliminación lógica

Siempre que sea posible utilizar:

isActive

en lugar de borrar registros.

Esto permite:

auditoría

historial

recuperación

trazabilidad

---

# DECISIÓN 009

## Auditoría

Todo módulo importante debe registrar:

createdAt

updatedAt

usuario responsable

fecha

hora

Nunca perder trazabilidad.

---

# DECISIÓN 010

## Usuarios

El sistema utiliza:

AppUserEntity

UserSessionManager

Roles

Permisos

No utilizar usuarios anónimos.

---

# DECISIÓN 011

## Escalabilidad

Cada módulo nuevo debe poder crecer sin modificar módulos anteriores.

Ejemplo:

Vacaciones

↓

Permisos

↓

Horas Extras

↓

Préstamos

↓

Evaluaciones

Todos siguen la misma arquitectura.

---

# DECISIÓN 012

## Organización del proyecto

Nunca mover archivos fuera de su carpeta.

database/

repository/

engine/

model/

ui/

navigation/

docs/

memory/

Mantener consistencia.

---

# DECISIÓN 013

## Documentación

Todo cambio importante debe actualizar:

CHANGELOG.md

y

OSINET_PROJECT_GUIDE.md

La documentación forma parte del proyecto.

---

# DECISIÓN 014

## Calidad

Nunca aceptar:

código duplicado

clases gigantes

lógica repetida

imports innecesarios

código muerto

Siempre refactorizar cuando sea necesario.

---

# DECISIÓN 015

## Filosofía del proyecto

OSINET Time ERP Enterprise NO es una aplicación sencilla.

Es un ERP empresarial.

Toda decisión debe favorecer:

estabilidad

mantenibilidad

escalabilidad

reutilización

rendimiento

legibilidad

---

# DECISIÓN 016

## Compatibilidad

Toda modificación debe mantener compatibilidad con el código existente.

No romper módulos anteriores.

Siempre ampliar.

Nunca destruir funcionalidad existente.

---

# DECISIÓN 017

## Inteligencia Artificial

Toda IA que participe en el proyecto debe actuar como:

Arquitecto Senior

Desarrollador Senior Android

Especialista en Kotlin

Especialista en Jetpack Compose

Especialista en Room Database

No debe generar ejemplos simples.

Debe producir código listo para producción.
